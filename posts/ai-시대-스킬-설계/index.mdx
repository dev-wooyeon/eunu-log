> 프롬프트는 요청이고, 스킬은 반복 가능한 팀 운영 방식이에요.

요즘 제가 가장 크게 체감하는 스킬은 사내 표준화 스킬이에요.
그중에서도 `Codex`에 넣어둔 테스트 코드 표준화 스킬이 가장 효과가 컸어요.

이전에도 AI로 테스트 코드는 빨리 만들 수 있었어요.
문제는 빨리 만든 테스트가 팀의 기준을 계속 흔든다는 점이었어요.

같은 기능을 검증하는데도 어떤 테스트는 `@SpringBootTest`를 올리고,
어떤 테스트는 `@WebMvcTest`를 쓰고,
어떤 테스트는 `assertEquals`와 `assertThat`을 섞고,
어떤 테스트는 수동 확인용 `@Disabled`만 남겨두고 끝났어요.

AI를 붙이자 속도는 빨라졌지만,
리뷰어가 다시 손으로 기준을 바로잡는 일도 같이 늘었어요.

그때부터 방향을 바꿨어요.
프롬프트를 더 잘 쓰는 게 아니라,
"우리 팀에서는 테스트를 이 기준으로 만든다"를
스킬로 고정하기 시작했어요.

이 글은 그 전환을 정리한 기록이에요.
스킬이 있으면 편하다는 이야기가 아니라,
실제로 어떤 기준을 문서화했고,
그 전후로 개발 편의성과 리뷰 품질이 어떻게 달라졌는지를 적어보려고 해요.

---

## Overview

AI 시대의 생산성 차이는 모델 성능보다
팀의 기준을 얼마나 호출 가능한 형태로 고정했는지에서 커져요.

프롬프트는 한 번의 요청으로 끝나기 쉬워요.
반면 스킬은 목적, 절차, 금지사항, 예외 조건, 리뷰 기준을 남겨요.
그래서 같은 작업을 여러 번 반복할 때 결과가 덜 흔들려요.

특히 테스트 코드는 이 차이가 크게 드러나요.
테스트는 코드이면서 문서이기 때문에,
형식과 판단 기준이 흔들리면 유지보수 비용이 바로 커져요.

---

## 문제 정의

테스트 코드를 생성하는 일 자체는 어려운 문제가 아니었어요.
문제는 생성된 테스트가 리뷰를 통과할 만큼 팀 기준에 맞느냐였어요.

스킬을 만들기 전에는 같은 요청을 자주 반복했어요.

- 기존 테스트 스타일을 먼저 읽어줘.
- 서비스 로직이면 Spring 컨텍스트 없이 테스트해줘.
- controller 계약이면 `@WebMvcTest`를 먼저 검토해줘.
- 새 테스트에서 Spock/Groovy는 추가하지 말아줘.
- `@MockBean`을 관성적으로 쓰지 말고 기본은 `@MockitoBean`으로 가줘.
- 테스트를 바꿨으면 관련 테스트도 직접 실행해줘.

이걸 한 번의 대화에서는 통제할 수 있었어요.
하지만 다음 작업으로 넘어가면 거의 다시 처음부터 설명해야 했어요.

그 결과는 비슷했어요.

- 서비스 단위 테스트인데 `@SpringBootTest`를 올려서 너무 무거워져요.
- unit, slice, integration 구분이 흐려져 실패 원인이 애매해져요.
- assertion 스타일과 mocking 스타일이 파일마다 섞여요.
- 수동 확인용 `@Disabled` 테스트가 목적 설명 없이 남아요.
- 리뷰가 버그나 회귀보다 스타일 교정에 더 많은 시간을 써요.

체감상 제일 답답했던 건 여기였어요.
AI가 코드를 빨리 써줘도,
리뷰어가 "이 테스트는 왜 이 타입인가",
"왜 여기서 Spring을 띄우는가",
"왜 `@MockBean`을 썼는가"를 다시 묻기 시작하면
실제 리드타임은 생각보다 줄지 않았어요.

그래서 문제를 이렇게 다시 정의했어요.

> 테스트 코드를 잘 생성하는 문제가 아니라,
> 팀의 테스트 기준을 일관되게 재현하게 만드는 문제.

---

## 해결 방법

해결책은 "잘 써달라"가 아니라
"무엇이 기본 표준이고 무엇이 예외인지"를 스킬에 박아두는 것이었어요.

실제로 `Codex`에 넣은 `test-standardization` 스킬은
단순 프롬프트 템플릿이 아니라 작은 운영 문서에 가까워요.

스킬을 만들 때는 아래 순서로 정리했어요.

1. 현재 저장소를 먼저 읽게 했어요.
`build.gradle`, `pom.xml`, `src/test`를 읽고
지금 어떤 테스트 스택이 섞여 있는지부터 파악하게 했어요.

2. 테스트를 네 종류로 강제 분류했어요.
대상 테스트를 `unit`, `slice`, `integration`, `manual` 중 하나로 먼저 나누게 했어요.
이 분류가 있어야 `@SpringBootTest`를 어디까지 허용할지 판단할 수 있어요.

3. 기본 표준을 명시했어요.
신규 테스트와 수정되는 테스트는
`Java + JUnit 5 + AssertJ + Mockito(BDDMockito)`로 통일하도록 했어요.
이 단계에서 이미 Spock/Groovy 신규 추가는 금지로 넣었어요.

4. 타입별 규칙을 분리했어요.
서비스 로직은 Spring 없이 검증하고,
controller 계약은 `@WebMvcTest`,
persistence는 `@DataJpaTest`를 우선 검토하게 했어요.
`@SpringBootTest`는 실제 런타임 조합이 검증 대상일 때만 쓰게 했어요.

5. 예외 규칙도 같이 적었어요.
Spring bean override가 필요하면 기본은 `@MockitoBean`,
특정 구성 제약 때문에 안 될 때만 예외적으로 `@MockBean`을 허용했어요.
중요한 건 "둘 다 가능"이 아니라
"기본은 하나이고 다른 하나는 예외"라고 박아두는 점이었어요.

6. 안티패턴을 문장으로 고정했어요.
서비스 단위 테스트에서 `@SpringBootTest` 사용,
일부 의존만 mock 한 채 unit test처럼 포장하는 방식,
목적 설명 없는 `@Disabled` 누적 같은 것들을 금지로 넣었어요.

7. 점진 이행 전략까지 같이 넣었어요.
repo 전체를 한 번에 갈아엎지 않고,
변경이 발생한 영역과 인접 테스트부터 표준을 적용하게 했어요.
실무에서는 이 부분이 특히 중요했어요.
표준이 좋아도 전면 교체 전략이면 바로 실패하니까요.

8. 마지막으로 PR 체크리스트를 붙였어요.
이 테스트가 왜 이 유형인지,
왜 `@SpringBootTest`가 필요한지,
`@MockitoBean`을 쓰지 못한 이유가 있는지,
관련 테스트를 실제로 실행했는지까지 확인하게 했어요.

지금 와서 보면 핵심은 간단해요.
좋은 스킬은 이상적인 정답을 적는 문서가 아니라,
기본 선택지, 예외 조건, 금지사항, 이행 전략을 함께 갖는 문서예요.

---

## 스킬 전후 차이

스킬을 만들기 전후의 차이는 "더 잘 쓴 테스트가 생겼다"보다
"어떤 대화를 안 하게 됐는가"에서 더 크게 느껴졌어요.

스킬 전에는 이런 식이었어요.

- AI가 초안을 빠르게 만들어요.
- 리뷰에서 테스트 타입, annotation, assertion 스타일을 다시 교정해요.
- 왜 이 테스트가 무거운지, 왜 이 mock 조합이 애매한지 다시 설명해요.
- 다음 작업에서 비슷한 교정을 또 반복해요.

스킬 후에는 흐름이 달라졌어요.

- 먼저 저장소 테스트 문맥을 읽고 현재 혼재 패턴을 파악해요.
- 대상 테스트를 어떤 유형으로 볼지 먼저 정해요.
- 그 분류에 맞는 annotation, mocking, assertion 스타일을 고릅니다.
- 리뷰에서는 스타일보다 누락 케이스와 회귀 위험을 먼저 봐요.

이 차이가 생각보다 컸어요.
예전에는 "형식이 맞는지"부터 다시 손봐야 했는데,
지금은 "실제로 이 시나리오가 빠졌는지"를 더 빨리 볼 수 있어요.

개발 편의성도 확실히 좋아졌어요.

- 매번 같은 기준을 설명하는 시간이 줄었어요.
- 초안의 품질 편차가 작아져서 첫 리뷰 진입 속도가 빨라졌어요.
- 신규 팀원이나 다른 작업에서도 같은 기준을 바로 불러올 수 있어요.
- 테스트 수정이 코드 작성 문제가 아니라 운영 기준 적용 문제로 바뀌었어요.

가장 중요한 변화는 이거였어요.
AI가 만든 결과물에 사람의 기준을 나중에 덧칠하는 방식에서,
처음부터 기준을 내장한 상태로 생성하게 만드는 방식으로 바뀌었다는 점이요.

이 전환 이후에는 테스트 PR에서
"이렇게도 쓸 수 있지 않나?" 같은 취향 토론이 줄고,
"이 테스트가 왜 이 유형인지", "무엇이 아직 검증되지 않았는지"를
더 빨리 이야기하게 됐어요.

---

## 결과

이 경험을 통해 확인한 건 세 가지예요.

첫째, 스킬은 개인 프롬프트 자산이 아니라 팀 운영 자산이에요.
좋은 스킬은 "잘 써줘"를 잘게 쪼갠 문장이 아니라,
팀의 기본 선택지를 코드처럼 고정한 문서예요.

둘째, 깊이 있는 스킬은 항상 예외와 금지사항을 같이 가져요.
기본은 `@MockitoBean`이지만 언제 `@MockBean`이 예외인지,
`@SpringBootTest`를 언제만 써야 하는지,
왜 repo 전체 일괄 변환을 피하는지가 같이 적혀 있어야 해요.
이게 없으면 스킬은 보기 좋은 원칙집으로 끝나요.

셋째, 스킬의 진짜 효용은 결과물의 모양보다 리뷰 대화의 질에서 드러나요.
스타일 토론이 줄고,
누락된 케이스와 회귀 위험을 먼저 보게 된다면
그 스킬은 실제로 팀의 개발 편의성을 올리고 있는 거예요.

그래서 요즘은 AI 활용을 이렇게 봐요.

> 좋은 결과물을 한 번 뽑는 것보다,
> 같은 기준의 결과물을 반복해서 뽑게 만드는 쪽이 더 중요해요.

프롬프트는 순간의 성과를 만들 수 있어요.
하지만 스킬은 팀이 다시 쓸 수 있는 기준을 남겨요.
제가 지금 가장 높게 평가하는 스킬이
테스트 코드 표준화 스킬인 이유도 여기에 있어요.

## 실제로 공유하는 `test-standardization`

아래는 제가 실제로 `Codex`에 넣어둔
`test-standardization` 스킬이에요.

```markdown
---
name: test-standardization
description: "Use when the task is to define, review, or apply a shared test-code standard in a Java/Spring repository, especially to separate unit, slice, integration, and manual tests, reduce unnecessary @SpringBootTest usage, or migrate mixed Spock/JUnit styles toward Java + JUnit 5 + AssertJ + Mockito."
---

# Test Standardization

이 스킬은 Java/Spring 저장소의 테스트 코드를 일관된 표준으로 정리할 때 사용한다.
특히 아래 상황에 맞춘다.

- 테스트 표준 문서를 만들거나 다듬고 싶을 때
- `@SpringBootTest` 남용을 줄이고 싶을 때
- unit test와 integration test의 경계를 다시 잡고 싶을 때
- Spock/Groovy와 JUnit/Java가 섞인 테스트를 정리하고 싶을 때
- 신규 기능 개발과 기존 수정 시점에 맞춘 점진 이행 전략이 필요할 때

## 언제 사용하나

다음과 같은 요청에서 사용한다.

- "테스트 코드 표준 문서 정리해줘"
- "Spock 테스트를 JUnit5로 옮기자"
- "`@SpringBootTest`를 줄이고 테스트 타입을 분리하자"
- "이 저장소 테스트 스타일을 표준화하자"
- "테스트 작성 규칙과 PR 체크리스트를 만들자"

## 핵심 워크플로우

1. `build.gradle`, `pom.xml`, `src/test`를 읽어 현재 테스트 스택과 혼재 패턴을 파악한다.
2. 대상 테스트를 `unit`, `slice`, `integration`, `manual` 중 하나로 분류한다.
3. 기본 표준은 `Java + JUnit 5 + AssertJ + Mockito(BDDMockito)`로 잡는다.
4. 서비스 단위 로직은 Spring 컨텍스트 없이 검증하도록 유도한다.
5. API, persistence, runtime integration은 목적에 맞는 슬라이스 또는 통합 테스트로 분리한다.
6. Spring bean override가 필요한 테스트는 기본적으로 `@MockitoBean`으로 통일한다.
   Spring Boot 3.4+ 저장소를 기본 대상으로 보고, 특정 테스트 구성이나 호환성 문제로 `@MockitoBean`이 성립하지 않을 때만 예외적으로 `@MockBean`을 사용한다.
7. 대규모 일괄 변경 대신 변경이 발생한 영역을 중심으로 점진 이행 계획을 제안한다.
8. 테스트를 바꿨다면 관련 대상 테스트를 직접 실행해 행동 보존을 확인한다.

## 기본 원칙

- 신규 테스트와 수정되는 테스트는 Java 기반으로 통일한다.
- unit test는 실제 Bean 주입 없이 작성한다.
- controller 계약 검증은 `@WebMvcTest`를 우선 사용한다.
- persistence 검증은 `@DataJpaTest`를 먼저 검토한다.
- `@SpringBootTest`는 실제 런타임 조합이 검증 대상일 때만 사용한다.
- Spring bean override 기본 표준은 `@MockitoBean`이다.
- `@MockBean`은 `@MockitoBean`으로 해결되지 않는 예외 케이스에서만 사용한다.
- 수동 확인용 테스트는 `@Disabled`와 고정 문자열 prefix로 의도를 드러낸다.
- 테스트 유형을 드러내고 싶다면 짧은 prefix를 사용한다.
  예: `단위:`, `슬라이스:`, `통합:`, `수동:`

## 문서화 규칙

테스트 표준 문서나 제안서를 작성할 때는 아래 순서를 기본으로 쓴다.
다만 사용자가 문서를 명시적으로 요청하지 않았다면, 이 규칙은 내부 작업 기준으로만 사용하고
새 문서를 기본으로 생성하지 않는다.

1. 목적
2. 배경과 현재 문제
3. 기대 효과
4. 리스크와 점진 이행 전략
5. 표준 결정
6. 테스트 타입별 규칙
7. 금지 및 지양 항목
8. PR 체크리스트

## 코드 변경 규칙

- repo 전체를 한 번에 변환하려 하지 않는다.
- touch 하는 기능과 인접 테스트부터 표준을 적용한다.
- 스타일 통일보다 행동 보존과 회귀 방지가 우선이다.
- 한 파일 안에서 assertion 스타일과 mocking 스타일을 섞지 않는다.
- `System.out.println`, `@Disabled` 수동 테스트, 임시 TODO를 정리 대상으로 본다.

## 출력 기대치

요청에 따라 아래 중 하나를 만든다.

- 테스트 표준 문서
- 현재 저장소 테스트 진단 결과
- 점진 이행 계획
- PR 리뷰 코멘트
- 특정 테스트 파일의 JUnit5/AssertJ/Mockito 전환안

문서를 요청받지 않았다면 기본 출력은 코드 변경과 검증 결과를 우선한다.

- 일부 포트만 mock 한 채 실제 Bean 주입을 unit test처럼 사용하는 방식

- 테스트 문서를 썼지만 실제 코드 변경 기준과 연결되지 않는 상태

```

이걸 공유하는 이유는 단순해요.

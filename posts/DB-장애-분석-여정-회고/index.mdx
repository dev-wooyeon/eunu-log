짧은 기간 안에 비슷한 DB 장애가 두 번 반복됐다.
처음에는 애플리케이션 코드나 커넥션 풀 설정을 먼저 의심했다.
응답 지연, CPU 급등, 타임아웃 같은 현상은 늘 코드와 인프라부터 떠올리게 하기 때문이다.

하지만 슬로우 쿼리를 따라가며 결론이 달라졌다.
이번 문제는 "코드가 비효율적이었다"보다
"조회 패턴을 받쳐줄 인덱스가 없었다"에 더 가까웠다.

특히 `rows_examined`가 70만에 가까운 쿼리를 본 순간 병목의 방향이 완전히 바뀌었다.
이 글에서는 장애 상황에서 어떻게 원인을 좁혀갔는지,
왜 `avg_time`보다 `rows_examined`를 더 중요하게 봤는지,
그리고 인덱스를 어떤 기준으로 설계해야 하는지 정리한다.

---

## 1. 왜 코드부터 의심했는가

당시 보였던 현상은 전형적인 장애 징후였다.

- API 응답 지연
- DB CPU 급등
- 커넥션 풀 대기 증가
- 타임아웃 발생

처음에는 자연스럽게 애플리케이션 레이어를 먼저 봤다.
최근 배포 코드, 커넥션 풀 설정, 재시도 로직, 타임아웃 설정을 우선 점검했다.

그런데 로그를 모아보니 공통점이 하나 있었다.

> 느린 요청의 뒤에는 항상 슬로우 쿼리가 있었고,  
> 그 슬로우 쿼리의 뒤에는 풀스캔이 있었다.

이 시점부터 문제를 "코드 최적화"가 아니라
"왜 이 쿼리가 인덱스를 타지 못했는가"로 다시 정의하기 시작했다.

**정리:** 장애 증상은 애플리케이션에서 보였지만, 원인 후보는 DB 실행 경로에 더 가까웠다.

---

## 2. 문제를 어떻게 정의했는가

무작정 튜닝부터 하지는 않았다.
먼저 이번 대응의 우선순위를 정했다.

| 순서 | 목표 | 이유 |
| --- | --- | --- |
| 1 | 장애 원인 제거 | 반복 장애를 멈추는 게 가장 먼저였다 |
| 2 | 장애 전파 차단 | 느린 쿼리가 다른 요청까지 밀어내고 있었다 |
| 3 | 로그와 추적 개선 | 다음 장애 때 더 빨리 좁힐 기준이 필요했다 |
| 4 | 테스트 보강 | 재현 가능한 검증 장치가 필요했다 |
| 5 | 레거시 정리 | 근본 원인 제거 뒤에야 순서가 됐다 |

이 기준에서 가장 먼저 잡은 건 인덱스였다.
장애가 난 시점에 가장 큰 비용을 만든 건 애플리케이션 한 줄보다,
DB가 읽어야 했던 행 수 자체였기 때문이다.

---

## 3. 슬로우 쿼리에서 무엇을 봤는가

먼저 `performance_schema`에서 평균 실행 시간이 긴 쿼리들을 모았다.

```sql
SELECT
    DIGEST_TEXT,
    COUNT_STAR AS exec_count,
    ROUND(SUM_TIMER_WAIT / 1000000000000, 3) AS total_time_sec,
    ROUND(AVG_TIMER_WAIT / 1000000000000, 3) AS avg_time_sec,
    SUM_ROWS_EXAMINED AS rows_examined,
    SUM_ROWS_SENT AS rows_sent
FROM performance_schema.events_statements_summary_by_digest
ORDER BY AVG_TIMER_WAIT DESC
LIMIT 20;
```

여기서 중요했던 건 `avg_time_sec` 하나만 보는 게 아니었다.

| 지표 | 왜 봤는가 |
| --- | --- |
| `avg_time_sec` | 요청 하나가 얼마나 느린지 보여준다 |
| `exec_count` | 느린 쿼리가 얼마나 자주 호출되는지 보여준다 |
| `rows_examined` | DB가 실제로 얼마나 많이 읽었는지 보여준다 |
| `rows_sent` | 읽은 결과 중 실제로 반환된 양을 보여준다 |

문제 쿼리 하나는 형태가 아주 단순했다.

```sql
WHERE unique_id = ?
```

그런데 지표는 단순하지 않았다.

- 평균 실행 시간: 약 0.6초
- `rows_examined`: 약 740,000
- `rows_sent`: 거의 없음

이 숫자를 문장으로 바꾸면 이렇게 된다.

> 데이터 한 건을 찾기 위해,  
> 매번 74만 건 가까이를 뒤지고 있었다.

모든 슬로우 쿼리가 인덱스 문제는 아니다.
정렬, 조인 순서, 통계 정보, 락 경합이 원인일 수도 있다.
하지만 이번 사례처럼 `rows_examined`가 비정상적으로 크고
`rows_sent`는 거의 없는 경우에는,
"필요한 것보다 훨씬 많이 읽고 있다"는 신호가 아주 분명했다.

**정리:** DB 비용은 결과보다 읽은 양에서 먼저 커진다.

## 4. 인덱스 현황과 실행계획을 확인했다

슬로우 쿼리를 본 뒤에는
"이 컬럼에 인덱스가 원래 있었는가"를 바로 확인했다.

```sql
SELECT
    TABLE_NAME,
    INDEX_NAME,
    GROUP_CONCAT(COLUMN_NAME) AS indexed_columns
FROM INFORMATION_SCHEMA.STATISTICS
WHERE TABLE_SCHEMA = 'tableName'
GROUP BY TABLE_NAME, INDEX_NAME;
```

문제 테이블인 `activity_queue`는 사실상 기본키만 있었다.

| INDEX_NAME | indexed_columns |
| --- | --- |
| `PRIMARY` | `seq` |

문제 쿼리는 `unique_id`로 찾고 있었는데,
인덱스는 `seq`에만 걸려 있었던 것이다.

즉, 이번 조회 패턴 기준으로는
기본키 인덱스가 있어도 아무 도움이 되지 않았다.

이후 `EXPLAIN`으로 실행계획을 확인했다.

```sql
EXPLAIN
SELECT ...
FROM activity_queue
WHERE unique_id = ?;
```

결과는 아래처럼 나왔다.

```text
type = ALL
key = NULL
```

이건 꽤 명확한 신호였다.

- `type = ALL`: 전체 행을 훑고 있다
- `key = NULL`: 선택된 인덱스가 없다

즉, 추측이 아니라 실행계획 수준에서 풀스캔이 확인된 상태였다.

**정리:** 슬로우 쿼리 분석은 의심의 출발점이고, `EXPLAIN`은 확정의 단계다.

## 5. 인덱스는 컬럼이 아니라 쿼리 기준으로 설계해야 한다

이 경험에서 가장 크게 남은 기준은 여기에 있다.

> 인덱스는 컬럼 이름을 보고 추가하는 게 아니라,  
> 실제 호출되는 쿼리 패턴을 보고 설계해야 한다.

당시 자주 호출되던 조건은 대략 두 가지였다.

```sql
WHERE unique_id = ?
```

```sql
WHERE unique_id = ? AND type = ? AND status = ?
```

이 패턴을 보면 중요한 건 컬럼 목록이 아니라
"어떤 순서로 동등 조건이 붙는가"다.

처음 떠올릴 수 있는 후보는 이런 식이었다.

```sql
(unique_id)
(unique_id, type, status)
```

다만 여기서 멈추면 안 된다.
두 인덱스를 모두 추가하는 게 항상 정답은 아니기 때문이다.
복합 인덱스는 왼쪽 prefix를 활용할 수 있어서,
`(unique_id, type, status)` 하나만으로도 첫 번째 조회를 상당 부분 커버할 수 있다.

그래서 실제 선택은 아래 기준을 같이 봐야 한다.

| 확인 기준 | 왜 필요한가 |
| --- | --- |
| 실행계획 변경 여부 | 정말 풀스캔이 인덱스 조회로 바뀌는지 확인해야 한다 |
| 카디널리티 | 값이 너무 치우치면 기대만큼 효과가 안 날 수 있다 |
| 쓰기 비용 | 인덱스가 늘면 insert/update 비용도 같이 는다 |
| 다른 조회 패턴 | 지금 쿼리 하나만 최적화하면 다른 경로가 손해볼 수 있다 |

핵심은 "컬럼이 보이니 인덱스를 하나 더 만든다"가 아니다.
실행계획이 어떻게 바뀌고, 그 대가가 무엇인지까지 같이 봐야 한다.

**정리:** 좋은 인덱스는 컬럼 나열이 아니라 쿼리 패턴과 비용의 균형으로 결정된다.

## 6. 이 일은 DBA와 백엔드의 경계를 다시 생각하게 했다

예전에는 인덱스를 운영 이슈가 터진 뒤에 보는 영역처럼 여긴 적도 있었다.
그런데 이번 일을 겪고 보니 그 구분은 실무에서 그리 유용하지 않았다.

장애가 발생한 순간에는 DBA처럼 실행계획과 읽기 비용을 봐야 했고,
문제를 예방하는 단계에서는 백엔드처럼 API 호출 패턴과 조회 조건을 이해해야 했다.

둘 중 하나만으로는 부족했다.

- 실행계획만 보면 왜 이 쿼리가 자주 호출되는지 놓치기 쉽다
- 애플리케이션 코드만 보면 DB가 실제로 얼마나 많이 읽는지 놓치기 쉽다

결국 인덱스는 운영 이슈가 아니라 설계 이슈이기도 했다.
느려진 뒤에 고치는 일과, 느려지지 않게 만드는 일은 분리되지 않았다.

---

## 7. 이번 경험 이후로 바뀐 기준

이후에는 쿼리와 인덱스를 따로 보지 않으려고 한다.
지금은 아래 기준을 기본값으로 두고 있다.

1. 새로운 조회 쿼리를 만들 때는 인덱스 후보를 같이 적는다.
2. 배포 전 주요 쿼리는 `EXPLAIN`으로 한 번 더 확인한다.
3. 슬로우 쿼리 모니터링은 평균 시간보다 `rows_examined`까지 같이 본다.
4. 인덱스를 추가할 때는 읽기 이득과 쓰기 비용을 같이 기록한다.
5. 자주 쓰는 조회 패턴은 팀 문서에 남겨 다음 설계의 기준으로 삼는다.

특히 "기본키가 있으니 괜찮다"는 생각은 완전히 버리게 됐다.
기본키 인덱스는 당연히 중요하지만,
문제 쿼리의 조건과 맞지 않으면 그 쿼리에는 없는 것과 크게 다르지 않았다.

## 8. 마무리

이번 장애는 겉으로 보면 API 장애였다.
하지만 실제로는 애플리케이션 코드보다,
쿼리와 인덱스 사이의 약속이 깨진 문제에 가까웠다.

특히 `rows_examined`를 본 뒤에는 판단 기준이 선명해졌다.
DB는 결과를 반환하는 비용보다,
그 결과를 찾기 위해 얼마나 많이 읽었는지에서 먼저 무거워졌다.

지금은 이렇게 정리한다.

> DB 성능 문제는 코드 한 줄보다,  
> 쿼리 패턴과 인덱스 설계가 어긋나는 순간부터 시작되는 경우가 많다.

다음에는 장애가 나서 인덱스를 보는 게 아니라,
쿼리를 작성하는 순간부터 어떤 실행계획을 원하고 있는지 먼저 생각하려고 한다.

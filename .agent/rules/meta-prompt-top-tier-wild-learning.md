# Meta Prompt: Top-Tier 설명 구조 + Wild Learning 실행 규칙

## 목적

이 문서는 미경험 영역(예: 행동로그 수집/분석)에서도
빠르게 실무 성과를 만들고, 동시에 깊은 이해를 축적하기 위한 규칙이다.
항상 다음 두 축을 함께 적용한다.

1. 탑티어 기술블로그식 설명 구조
2. 야생형 Top-Down 학습/실행 루프

## 적용 범위

- 설계 제안
- 코드 구현
- 리뷰/회고
- 문서화
- 기술블로그 작성

## 설명 구조 강제 규칙 (항상 이 순서)

모든 응답/문서/PR 설명은 아래 구조를 따른다.

1. 결론 먼저
- 무엇이 얼마나 개선되는지 수치로 먼저 제시한다.
- 수치가 없으면 "현재 측정 불가"를 명시하고, 측정 태스크를 최우선으로 둔다.

2. 문제 정량화
- 현재 상태의 비용/리스크를 수치로 못 박는다.
- 예: 누락률, 지연 시간, 전환률, 장애 빈도, 운영 시간, 추정 손실.

3. 대안 비교 (최소 2~3개)
- 대안별 장단점, 비용, 운영 난이도, 확장성, 리스크를 비교한다.
- 최종 선택 이유와 기각 이유를 모두 기록한다.

4. 구현은 원리 중심
- 코드 나열보다 의도, 메커니즘, 트레이드오프를 설명한다.
- "왜 이렇게 설계했는지"를 컴포넌트/데이터 흐름 단위로 설명한다.

5. 한계와 후속 과제 명시
- 현재 접근의 한계, 미해결 리스크, 다음 실험 항목을 숨기지 않는다.

## 출력 포맷 템플릿

아래 골격을 기본 템플릿으로 사용한다.

```md
## 결론
- 개선 결과: [지표] [기존] -> [변경] ([증감률])

## 문제 정의(정량)
- 현재 비용/리스크: ...
- 근거 데이터: ...

## 대안 비교
- A안: ... (장점/단점/리스크)
- B안: ... (장점/단점/리스크)
- C안: ... (선택사항)
- 선택: [채택안], 이유: ...

## 구현 원리
- 핵심 메커니즘:
- 설계 트레이드오프:
- 검증 방법:

## 한계와 다음 단계
- 한계:
- 후속 과제:
```

## Wild Learning 실행 규칙 (Top-Down)

기초부터가 아니라 결과물부터 시작한다.
"동작하는 데모 -> 원리 역추적 -> 일반화" 순서를 고정한다.

1. 데모 우선
- 이력서 문장보다 데모 링크를 우선 만든다.
- "클릭 1번으로 실력 검증" 가능한 산출물을 목표로 한다.

2. 프로젝트 선실행
- 실무 문제를 바로 구현한다.
- 막히는 개념은 구현 중 필요한 만큼만 즉시 학습한다.

3. 재귀적 갭 채우기
- 모르는 개념은 AI에 연쇄 질문으로 끝까지 파고든다.
- 이해가 끊기면 절대 다음 단계로 넘어가지 않는다.

4. 실패 기록 의무화
- 성공 경로만 남기지 말고 실패한 대안/가설도 기록한다.
- "왜 안 됐는지"가 다음 의사결정의 자산이다.

5. 학습 완료 기준
- 설명 가능(남에게 5분 내 설명)
- 재현 가능(새 환경에서 다시 구현)
- 응용 가능(유사 문제에 변형 적용)

## AI 질의 패턴 (실전용)

미경험 영역에서 아래 질문을 기본 패턴으로 사용한다.

1. "12살에게 설명하듯 비유로 설명해줘."
2. "왜 이 방법을 선택했고, 대안은 왜 기각했어?"
3. "중간 과정(데이터 흐름/shape/상태 변화)을 단계별로 보여줘."
4. "이 접근의 실패 조건과 관측 신호는 뭐야?"
5. "기존 방식 대비 정확히 달라진 점만 리스트로 정리해줘."

## 미경험 도메인(행동로그) 안전장치

행동로그는 눈에 바로 보이지 않는 영역이므로,
아래 검증 루프를 의무 적용한다.

1. 이벤트 사전 정의
- 이벤트명, 파라미터, 타입, 예시 payload, owner를 먼저 고정한다.

2. 계측 전/후 검증 시나리오
- 로컬 이벤트 발생 -> 수집 API 도달 -> 저장 -> 대시보드 노출까지
  end-to-end 체크리스트를 만든다.

3. 품질 지표 상시 모니터링
- 누락률, 중복률, 지연률, 스키마 불일치율을 최소 지표로 운영한다.

4. 개인정보/컴플라이언스
- PII 금지 필드를 명시하고 수집 단계에서 차단한다.
- 동의(Consent) 상태를 이벤트와 함께 관리한다.

5. 단계적 확장
- `page_view / post_view / scroll_depth / cta_click`부터 시작하고
  안정화 후 이벤트를 확장한다.

## 의사결정 필터

- 내 목표와 인센티브가 맞는 조언만 채택한다.
- "안전하지만 성장이 느린 선택"과 "위험하지만 학습 밀도가 높은 선택"을 구분해 기록한다.
- 환경 레버리지를 고려해, 밀도 높은 커뮤니티/피드백 루프에 지속적으로 노출된다.

## 금지 규칙

- 수치 없는 "좋아졌다" 표현 금지
- 대안 비교 없는 단일안 강행 금지
- 원리 설명 없는 코드 덤프 금지
- 한계/리스크 은폐 금지

## 최종 목표

모든 산출물은 다음을 동시에 만족해야 한다.

1. 제품 가치 증명 (사용자/비즈니스 지표 개선)
2. 엔지니어링 깊이 증명 (설계/트레이드오프/운영 능력)
3. 학습 민첩성 증명 (미경험 영역을 실전으로 빠르게 흡수)

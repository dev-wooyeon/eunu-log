# Toss UX Principles (실행 요약판)

## 1) 글쓰기 원칙

- 짧고 쉬운 단어를 쓴다.
- 한 문장에는 한 메시지만 넣는다.
- 모호한 표현 대신 수치/조건을 쓴다.
- 버튼 문구는 동사로 시작한다.

예시:

- `오류가 발생했습니다` 대신
- `인터넷 연결이 불안정해요. 다시 시도해 주세요.`

## 2) 정보 구조

- 한 화면의 핵심 행동은 1개로 제한한다.
- 중요 정보 먼저, 상세 정보는 필요할 때 노출한다.
- 긴 흐름은 단계로 나눈다. (입력 → 확인 → 완료)

## 3) 시각적 위계

- 구분선보다 여백으로 그룹을 만든다.
- 간격 기본값: `8 / 16 / 32`
- CTA는 화면 내에서 가장 먼저 보이게 배치한다.

## 4) 인터랙션

- 피드백은 즉시 보여준다. (목표: 100ms 이내)
- Hover/Active는 명확하되 과하지 않게 적용한다.
- 로딩 상태는 반드시 표시한다. (스켈레톤/스피너)

## 5) 모션

- 의미 있는 모션만 사용한다.
- 기본 권장 시간:
  - 버튼/토글: `150~200ms`
  - 카드/리스트: `200~300ms`
  - 모달/시트: `300~400ms`
- 성능 원칙:
  - `transform`, `opacity` 중심
  - 60fps 유지

## 6) 모바일 우선

- 최소 터치 영역: `44x44px` (권장 `48x48px`)
- 인접 터치 요소 간격: `8px` 이상
- 자주 쓰는 행동은 하단 영역에 배치한다.

## 7) 접근성

- 대비:
  - 본문 4.5:1 이상
  - 큰 텍스트 3:1 이상
- 키보드 탭 이동 가능해야 한다.
- 상태 변화는 스크린리더에 전달되어야 한다.

## 8) 컴포넌트 기준

- 버튼 계층:
  - Primary: 주요 행동
  - Secondary: 보조 행동
  - Tertiary: 텍스트 행동
- 입력 필드:
  - 라벨 항상 노출
  - 에러는 원인 + 해결 방법을 같이 제공

## 9) 품질 체크리스트

- 다음 행동이 한눈에 보이는가?
- 빈 상태/로딩/에러가 모두 설계되었는가?
- 모바일에서 터치/가독성 문제가 없는가?
- 다크/라이트 모두 대비가 충분한가?
